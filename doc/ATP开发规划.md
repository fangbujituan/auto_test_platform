# ATP 自动化测试平台 — 开发规划

> 本文档描述 ATP 平台的中长期演进方向、阶段划分和技术选型。
> 作者: yandc | 创建时间: 2026-05

---

## 一、平台愿景

打造一个**AI 驱动的全栈自动化测试平台**，覆盖 API、WebUI、AppUI、性能测试四大领域，通过 AI 智能生成测试数据和用例，实现高效化、智能化测试。

**AI 在平台中承担两个角色：**

| 角色 | 职责 |
|------|------|
| 开发助手 | 辅助编写平台功能代码 |
| 测试数据引擎 | 自动生成测试用例、测试数据，写入数据库 |

### 第三阶段：AI 智能化

**目标：** AI 自动生成测试数据和用例，减少人工编写成本。

| 任务 | 技术选型 | 说明 |
|------|---------|------|
| MCP 集成 | Model Context Protocol | 让 AI 能调用平台内部接口（创建用例、查询接口等） |
| LangChain 编排 | LangChain | 多步骤 AI 任务的流程控制和工具调用 |
| 测试数据生成 | AI + MCP | 根据接口定义自动生成测试用例和边界数据 |
| 智能断言推荐 | AI | 根据接口响应结构推荐断言规则 |
| 执行结果分析 | AI | 分析失败原因，推荐修复方案 |

**AI 工作流示例：**

```
用户触发"智能生成用例"
    ↓
LangChain 编排：
    1. 调用 MCP 工具：获取接口定义（method、path、参数）
    2. 调用 AI：根据接口定义生成测试用例（正常、异常、边界）
    3. 调用 MCP 工具：校验用例格式
    4. 调用 MCP 工具：写入数据库
    5. 如果格式不合格 → 重新生成
    ↓
用例自动入库，用户可直接执行
```

**LangChain 简介：**
LangChain 是一个编排 AI 调用的框架。它解决的核心问题是：AI 调用不是一次就够的，需要多轮对话、工具调用、结果校验、重试。LangChain 把这些编排逻辑标准化。

- **MCP** = AI 能调用的工具集（创建用例、查询接口、执行测试等）
- **LangChain** = 编排 AI 什么时候调用哪个工具、怎么处理结果

## 四、补充建议

### 4.1 插件化架构

随着引擎越来越多，建议引入插件机制：

```python
# 引擎注册
engine_registry = {
    "api": ApiEngine,
    "webui": WebUIEngine,
    "appui": AppUIEngine,
    "performance": PerformanceEngine,
}

# 统一调用
engine = engine_registry[test_type]()
engine.execute(case_data)
```

好处：新增引擎不需要改核心代码，只需注册。

### 4.2 异步执行队列

当前执行是同步的，随着用例量增大和引擎类型增多，建议引入任务队列：

| 方案 | 说明 |
|------|------|
| Celery + Redis | Python 标准异步任务方案 |
| APScheduler（当前） | 适合定时任务，不适合大量并发 |

### 4.3 测试环境管理增强

| 功能 | 说明 |
|------|------|
| 环境快照 | 保存/恢复环境配置 |
| 环境克隆 | 一键复制环境配置 |
| 环境健康检查 | 定期检测环境可用性 |

### 4.4 版本管理

| 功能 | 说明 |
|------|------|
| 用例版本历史 | 记录用例修改历史，支持回滚 |
| 接口变更检测 | 对比 Swagger 文档变化，自动标记受影响用例 |

## 七、价值分析

| 维度 | 价值 |
|------|------|
| 对个人 | 全栈自动化 + AI 工程实践，技术深度和广度兼备 |
| 对团队 | 统一测试平台，降低多工具切换成本 |
| 对产品 | AI 生成用例降低人工成本，提升测试覆盖率 |
| 商业化 | 具备 SaaS / 私有化部署能力，可对外输出 |

**风险提示：**
- 范围过大容易分散精力，建议每个阶段聚焦 1-2 个核心目标
- AI 生成质量依赖 prompt 工程和数据质量，需要持续迭代
- WebUI/AppUI 引擎维护成本高（浏览器/设备兼容性），建议先做 API 引擎做扎实
