# SAP 技术架构与产品设计分析

> 从技术架构和产品设计角度分析 SAP，提炼对我们 SaaS 产品建设有参考价值的设计思想。

---

## 一、数据库层：SAP HANA 的技术壁垒

SAP 最核心的技术资产是 HANA 数据库。它是一个内存列式关系型数据库，把 OLTP（事务处理）和 OLAP（分析处理）合并在一个引擎里（所谓 OLTAP）。

关键设计点：
- 数据常驻内存（RAM），不依赖磁盘 I/O，查询速度比传统数据库快几个数量级
- 列式存储，天然适合聚合分析，压缩率高
- 列式存储也天然支持多核并行处理，不同列的操作可以分配到不同 CPU 核心
- 不需要传统数据库的索引、物化视图、预聚合表等优化手段

**对我们的启发**：不需要自研数据库，但 SAP 的思路是"用底层技术优势绑定上层应用"。如果我们的产品能在某个技术点上做到足够深（比如测试执行引擎、智能用例生成），就能形成类似的技术壁垒。

## 二、部署模型：三种形态覆盖所有客户

SAP S/4HANA 提供三种部署模式，直接对应不同客户群体的需求：

| 部署模式 | 对应 SAP 产品 | 商业包装 | 特点 |
|---------|-------------|---------|------|
| 公有云（多租户 SaaS） | S/4HANA Cloud Public Edition | GROW with SAP | 标准化流程，不可深度定制，按季度自动升级 |
| 私有云（单租户托管） | S/4HANA Cloud Private Edition | RISE with SAP | 可定制，SAP 或合作伙伴托管，客户有更多控制权 |
| 本地部署 | S/4HANA On-Premise | 传统许可证 | 完全自主，永久许可 + 年度维护费 |

关键设计思想：

1. **公有云版本故意限制定制能力**，强制客户使用标准流程，这样才能做到多租户共享基础设施、统一升级
2. **私有云版本允许定制**，但通过"RISE with SAP"包装成订阅服务（OpEx 而非 CapEx），SAP 仍然掌控运维
3. **本地部署是传统模式**，给那些绝对不能上云的客户（金融、政府等）

**对我们的启发**：除了"SaaS 版 + 私有化版"两种形态，可以再加一个中间态——"托管私有云"，即我们帮客户运维，但数据在独立环境中。这样既满足了数据隔离需求，又不用把镜像交出去。

## 三、Clean Core 架构：扩展性设计的精髓

这是 SAP 近几年最重要的架构理念，对 SaaS 产品设计非常有参考价值。

核心原则：**不允许修改核心代码，只能通过标准 API 和扩展点来扩展。**

```
┌──────────────────────────────────────┐
│         客户自定义扩展                 │
│    (Side-by-Side Extension on BTP)   │
├──────────────────────────────────────┤
│         扩展点 / Released APIs        │  ← 唯一允许的交互边界
├──────────────────────────────────────┤
│         SAP 标准核心（Clean Core）     │  ← 不可修改
│    S/4HANA 标准业务流程和数据模型       │
└──────────────────────────────────────┘
```

SAP 把扩展分成了层级：

- **Tier 1（ABAP Cloud）**：首选方式，使用受限的云端 ABAP，只能调用已发布的 API
- **Tier 2（API 桥接层）**：把旧的自定义代码包装成标准 API，供 Tier 1 调用
- **Side-by-Side 扩展**：在 BTP 平台上独立开发微服务，通过 API 与核心系统交互

**对我们的启发**：如果产品未来要支持客户定制，一定要从现在就设计好扩展机制：
- 定义清晰的 API 边界，核心逻辑不允许客户修改
- 提供 webhook / 插件机制让客户扩展行为
- 这也是保护源码的一种架构手段——客户只能通过 API 交互，看不到内部实现

## 四、BTP 平台：PaaS 化的产品矩阵

SAP BTP（Business Technology Platform）是 SAP 的 PaaS 层，把数据管理、分析、AI、集成、自动化统一在一个平台上。

```
SAP BTP 五大支柱：
┌─────────┬──────────┬──────────┬──────────┬──────────┐
│ 数据库与  │  分析与   │  应用开发  │  自动化   │  AI 与   │
│ 数据管理  │  BI      │  与集成   │          │ 智能技术  │
└─────────┴──────────┴──────────┴──────────┴──────────┘
                        │
              运行在多云基础设施上
           (AWS / Azure / GCP / 阿里云)
```

产品设计思路：

- **核心 ERP 是"记录系统"**（System of Record）——稳定、标准、不轻易改动
- **BTP 是"创新层"**——客户在上面构建差异化应用
- 两者通过标准 API 连接，核心系统保持干净

**对我们的启发**：测试平台未来可以考虑类似的分层——核心测试管理是稳定的基座，上面提供插件市场/工具箱让用户扩展能力（我们已经有 toolbox 模块，可以沿着这个方向发展）。

## 五、多租户设计：SAP 的实际做法

### 公有云版本（多租户）

- 多个客户共享同一套基础设施和应用实例
- 数据完全隔离（每个租户有独立的数据分区）
- 配置隔离（每个租户可以有不同的业务配置，但不能改代码）
- 统一升级（所有租户同时升级到新版本，按季度发布）

### 私有云版本（单租户）

- 每个客户一个独立实例
- 可以做代码级定制
- 升级由 SAP 运维团队协调，不是自动的

### 许可模式对比

| 维度 | 公有云 | 私有云 | 本地部署 |
|------|-------|-------|---------|
| 付费模式 | 订阅（OpEx） | 订阅（OpEx） | 永久许可 + 年维护费（CapEx） |
| 定制能力 | 低（仅配置） | 高（可改代码） | 完全自由 |
| 升级方式 | 自动、强制 | 协调升级 | 客户自主 |
| 运维责任 | SAP 全托管 | SAP 或合作伙伴 | 客户自己 |
| 数据位置 | SAP 数据中心 | 可选数据中心 | 客户机房 |

## 六、对我们产品的具体借鉴

### 产品演进路径

```
当前状态                    中期目标                     长期目标
─────────                 ─────────                   ─────────
单体 Flask 应用      →     多租户 SaaS + 私有化版    →    平台化
单数据库             →     租户级数据隔离             →    插件生态
Mock Token          →     JWT + License 授权        →    SSO/OAuth2
项目级权限           →     租户级 + 项目级权限        →    自定义角色引擎
```

### 从 SAP 学到的核心设计原则

1. **部署灵活性是企业客户的刚需**
   - 公有云、私有云、本地部署至少要支持两种
   - 不同部署形态对应不同的商业包装和定价策略

2. **Clean Core 思想**
   - 从现在就设计好扩展边界，核心不可修改，通过 API 扩展
   - 这既是架构原则，也是源码保护手段

3. **数据隔离是信任基础**
   - 多租户共享架构要让客户相信数据是安全的
   - 否则客户会选择私有化，增加我们的运维成本

4. **平台化思维**
   - 核心产品做"记录系统"，保持稳定
   - 创新和定制放在扩展层，通过 API/插件实现
   - 工具箱（toolbox）模块是平台化的雏形，值得持续投入

5. **商业包装与技术架构对齐**
   - SAP 用 GROW with SAP（公有云）和 RISE with SAP（私有云）区分客户群
   - 技术上的限制（公有云不可定制）是有意为之，服务于商业目标（降低运维成本、统一升级）

## 七、我们的三种部署形态规划

参考 SAP 的模式，我们可以规划如下：

| 我们的部署形态 | 对标 SAP | 目标客户 | 技术实现 |
|-------------|---------|---------|---------|
| SaaS 公有云版 | GROW with SAP | 中小团队、个人 | 多租户共享架构，标准功能 |
| 托管私有云版 | RISE with SAP | 中大型企业 | 独立实例，我方运维，可定制配置 |
| 本地私有化版 | On-Premise | 强合规需求企业 | Docker 镜像 + License 交付 |

这三种形态共用同一套代码基础，通过配置和部署方式区分，而不是维护三套代码。
