## 1. Safari 访问 `/api/docs/openapi.json` 中文乱码

### 现象

- 用 Safari 打开 `http://<host>:12048/api/docs/openapi.json`，接口描述、字段中文显示为 `æµ‹è¯•` 之类乱码。
- 同一地址在 Chrome 中正常。

### 原因

`flask-smorest` 自动生成的 OpenAPI 文档接口返回的响应头是 `Content-Type: application/json`，**没有显式声明 charset**。

- Chrome 遵循 RFC 8259，对 `application/json` 默认按 UTF-8 解码 → 正常。
- Safari 在缺少 `charset` 时会做编码嗅探，对中文短文本经常猜成 Latin-1 → 乱码。

项目里 `CustomJSONProvider(ensure_ascii=False)` 让中文以原始 UTF-8 字节输出（这是想要的），但 Safari 没拿到 `charset=utf-8` 这个明确信号，只能瞎猜。

### 验证

```bash
curl -I http://<host>:12048/api/docs/openapi.json
# 修复前：Content-Type: application/json
# 修复后：Content-Type: application/json; charset=utf-8
```

### 处理

`app/flask_app.py` 中已添加全局 `after_request` 钩子 `_register_json_charset`，对所有 `application/json` 响应自动追加 `; charset=utf-8`。

```python
def _register_json_charset(app):
    @app.after_request
    def _force_json_charset(response):
        content_type = response.content_type or ""
        if response.mimetype == "application/json" and "charset" not in content_type.lower():
            response.headers["Content-Type"] = "application/json; charset=utf-8"
        return response
```

后续新增任何返回 JSON 的接口都会自动生效，无需单独处理。

## 2. Docker 部署后如何查看接口日志

项目日志分三层四个渠道，按定位问题的优先级排列。

### 渠道 A：宿主机日志文件（最常用）

`docker-compose.yml` 把容器内 `/app/logs` 挂到了宿主机 `/opt/vplatform/auto_test_platform/logs`，文件按天切分，命名 `info-YYYY-MM-DD.log`。

```bash
# 实时追当天日志
tail -f /opt/vplatform/auto_test_platform/logs/info-$(date +%Y-%m-%d).log

# 按接口路径过滤
grep "POST /api/auth/login" /opt/vplatform/auto_test_platform/logs/info-*.log

# 找异常堆栈
grep -i "error\|exception\|traceback" /opt/vplatform/auto_test_platform/logs/info-2026-06-15.log

# 按 IP 过滤
grep "ip=192.168.1.100" /opt/vplatform/auto_test_platform/logs/info-2026-06-15.log
```

文件中包含两类内容：

- **接口流水**（`api.access` logger，由 `flask_app.py` 的 `_log_request_end` 输出）：
  ```
  POST /api/auth/login | ip=... | status=200 | duration=42.3ms | 入参={...} | 出参={...}
  ```
- **业务日志和异常堆栈**（业务代码中 `logging.getLogger(__name__)`）。

### 渠道 B：容器标准输出（实时观察用）

同样的内容也写到容器 stdout，`docker compose logs` 可看：

```bash
docker compose logs -f flask-app                          # 实时
docker compose logs --tail=200 flask-app                  # 最近 200 行
docker compose logs --since=10m flask-app                 # 最近 10 分钟
docker compose logs flask-app | grep "POST /api/auth"     # 配合过滤
```

> 渠道 A 和 B 内容基本一致。A 持久化、便于历史查询；B 实时方便，容器重启可能丢。

### 渠道 C：Gunicorn 访问日志

只在容器 stdout 中（`Dockerfile` 配置 `--access-logfile -` `--error-logfile -`），宿主机文件里没有。用来判断：

- 请求是否真的到了 Gunicorn（不是被 Nginx 拦了）
- worker 超时、重启、SIGTERM 事件

```bash
docker compose logs -f flask-app
```

### 渠道 D：Nginx & MySQL

```bash
docker compose logs -f nginx-frontend   # 反向代理是否收到请求
docker compose logs -f mysql-db         # 数据库连接、慢查询、初始化错误
docker compose logs -f                  # 所有服务汇总
```

### 推荐排查路径

1. 前端报错 → `docker compose logs nginx-frontend` 看请求是否到达 Nginx
2. 请求是否到 Flask → `docker compose logs flask-app` 看 Gunicorn access 行
3. 业务逻辑错误 → `tail -f /opt/vplatform/auto_test_platform/logs/info-$(date +%Y-%m-%d).log` 看入参出参和堆栈
4. DB 层异常 → `docker compose logs mysql-db`

### 注意事项

- 日志按天切但**没有自动清理**，建议加 cron 清 30 天前的：
  ```bash
  0 3 * * * find /opt/vplatform/auto_test_platform/logs -name "info-*.log" -mtime +30 -delete
  ```
