质量飞轮运转说明

一句话讲清楚： 我们围绕"需求 → 用例 → 执行 → bug → 回归"这条主线搭了一个能自转的飞轮。 每一步都封装成一个 service 函数，再通过三个出口同时暴露： HTTP 给前端、LangChain @tool 给内部 Agent、MCP Server 给外部 IDE Agent（Cursor / Claude Code 等）。 一份逻辑三个入口是这个项目最值得理解的设计。

1. 飞轮全景：闭环是怎么转的

飞轮的目标是让"被测系统的质量持续提升"自动化。下图是闭环的高层视图。 箭头方向就是数据流方向，每个节点都对应一组可独立替换的能力。

  class C,B store
  class G,X agent

关键点：

  • 用例和 bug 都落在数据库（橙色方块），不是只在内存中流转。这样飞轮可以中断、可以人工介入、可以多端协作。
  • 失败用例自动建 bug（虚线之外这一步是自动的）。bug 的 related_test_cases 字段直接关联用例，steps_to_reproduce 抄用例的 steps —— 修复后跑回归直接拿这条 bug 关联的用例就行。
  • 每个齿轮都是独立 service，可以单独触发也可以串起来跑。

2. 三出口架构：为什么这么设计

飞轮里每一个能力（生成用例、执行用例、问答 LLM…）都同时给三种调用者用。 service 层是唯一的逻辑实现，三个出口都是它的薄包装。

这样设计的好处：

  • 逻辑只写一份。 加一个新能力，只要写一个 service 函数 + 在三个出口各包一层（每层 10 行），三种调用方式自动具备。
  • 外部 Agent 接入零成本。 同事用 Cursor 写代码时，他的 Agent 通过 MCP 直接调用我们的ai_generate_and_save_test_cases， 生成的用例自动落到我们库里。下游飞轮继续转，不需要他做任何额外集成。
  • 测试只测 service。 写 unit test 直接对着 service 写，不用过 Flask、不用启动 LangGraph、不用拉 MCP。
具体落点（精简版，不一一列举）：
HTTP 出口：app/routes/ai_agent.py
@tool 出口：app/agents/tools/{testcase_tools, run_tools, case_tools}.py
MCP 出口：app/mcp_server/server.py，启动 python -m app.mcp_server

3. 第一齿轮：从需求到用例

把需求文本喂进去，得到一组结构化、可直接落库的测试用例。这是飞轮的起点，没有这一步后面都转不起来。

flowchart LR
  REQ["需求文本
(产品文档 / 用户故事 / API 描述)"]
  --> SVC[generate_test_cases
service]
  SVC --> LLM[llm_gateway.get_model
本地或网关]
  LLM --> P{输出是否合法
JSON?}
  P -- ✅ --> PARSE[PydanticOutputParser
校验字段]
  P -- ❌ --> RECOVER[正则抠 JSON
兜底解析]
  RECOVER --> PARSE
  PARSE --> RES["GeneratedTestCaseList
(cases[], summary)"]
  RES -.可选.-> SAVE[save_generated_cases]
  SAVE --> DB[(test_case_management)]

  classDef agent fill:#ebf8ff,stroke:#3182ce
  classDef store fill:#fef5e7,stroke:#dd6b20
  class SVC,LLM agent
  class DB store

核心设计：

  • 结构化输出契约。LLM 输出必须能解析成 GeneratedTestCaseList（pydantic schema）， 字段命名直接对齐 TestCaseManagement 表，可以一对一落库不用映射。
  • 双解析模式自动兜底。环境变量 TESTCASE_GEN_MODE=structured 走 OpenAI tool calling（GPT/Claude 这种大模型最稳）； =parser 走 prompt 注入 JSON Schema（小模型友好，是默认值）。
  • Prompt 可编辑。从数据库 ai_prompt_templates 表读，产品/测试可以在前端改提示词不需改代码。

4. 第二齿轮：从用例到 bug

读取一批用例 ID，依次执行。失败的用例自动写入 bug 表，bug 上自动关联回用例 ID。 这是飞轮闭合的关键 —— 没有这一步，前面生成多少用例都只是文档。

flowchart TB
  IN[case_ids + project_id] --> LOAD[查 test_case_management
过滤项目和有效性]
  LOAD --> LOOP{对每条用例}
  LOOP --> EXEC[CaseExecutorProtocol.execute]

  subgraph executors["执行器实现 (可插拔)"]
    direction LR
    V1["AIVerdictExecutor
(v1: LLM 当裁判)"]
    V2["PlaywrightExecutor
(v2: 真浏览器)"]
    V3["ApiExecutor
(v2: 真 HTTP)"]
  end
  EXEC -.可替换.-> V1
  EXEC -.可替换.-> V2
  EXEC -.可替换.-> V3

  V1 --> RESULT{status?}
  RESULT -- passed --> COUNT[计数 +1]
  RESULT -- failed --> BUG[BugReporter.report]
  RESULT -- blocked --> COUNT
  BUG --> WRITE[写 bugs 表
related_test_cases=用例id
steps_to_reproduce=用例steps
actual_result=AI 实际观察]
  WRITE --> DB[(bugs)]
  COUNT --> NEXT{还有用例?}
  WRITE --> NEXT
  NEXT -- 是 --> LOOP
  NEXT -- 否 --> REPORT["CaseRunReport
{total, passed, failed, bug_ids, results[]}"]

  classDef agent fill:#ebf8ff,stroke:#3182ce
  classDef store fill:#fef5e7,stroke:#dd6b20
  classDef alt fill:#f0e6ff,stroke:#6b46c1
  class EXEC,V1,BUG agent
  class DB store
  class V2,V3 alt

核心设计：

  • 执行器是协议（Protocol），不是固定实现。v1 用 LLM 当裁判（最便宜，零基础设施）， 把用例的 steps + expected 喂给 LLM 让它判 pass/fail。把 service 里 AIVerdictExecutor 换成 PlaywrightExecutor 或 ApiExecutor 就升级到真实执行，编排逻辑零改动。
  • 失败即建 bug 是默认行为。reporter_id 没传则自动取一个系统用户兜底（保证飞轮不卡住）。 bug 优先级直接由用例优先级映射（P0→critical，P1→high，等等）。
  • 结果可序列化。CaseRunReport.to_dict() 直接返回给 HTTP / MCP，外部 Agent 拿到 bug_id 列表就能继续触发下游流程。
真实跑过的飞轮： 一次实测中 LLM 正确识别出"步骤是退出登录但预期是停留登录态"这种逻辑矛盾， 自动建了 bug，related_test_cases 关联到用例，actual_result 写入 LLM 观察到的 "未跳转到首页并显示用户已登录的状态"。 飞轮真转起来了。

5. 端到端示例：外部 Agent 触发完整闭环

假设同事在 Cursor 里写完一个新功能，他的 IDE Agent 想自动生成用例并执行回归。下面是这一次完整请求的时序。

sequenceDiagram
  autonumber
  participant Ext as 外部 Agent
(Cursor)
  participant MCP as MCP Server
(stdio 接入)
  participant Svc as service 层
  participant LLM as 本地 LLM
(192.168.1.7)
  participant DB as MySQL

  Ext->>MCP: ai_generate_and_save_test_cases
(requirement, project_id)
  MCP->>Svc: generate_test_cases_with_db_prompt
  Svc->>DB: 读 ai_prompt_templates
  DB-->>Svc: 系统提示词
  Svc->>LLM: chat completion (含 schema)
  LLM-->>Svc: JSON 用例列表
  Svc->>Svc: PydanticOutputParser 解析
  Svc->>DB: INSERT test_case_management × N
  DB-->>Svc: case_ids
  MCP-->>Ext: {summary, saved_cases:[...]}

  Note over Ext,DB: ─ 第一齿轮完成 ─

  Ext->>MCP: run_test_cases
(case_ids, project_id)
  MCP->>Svc: execute_cases
  loop 对每条用例
    Svc->>LLM: AIVerdictExecutor.execute
    LLM-->>Svc: {status, actual_result}
    alt status == failed
      Svc->>DB: INSERT bugs
(related_test_cases=[id])
      DB-->>Svc: bug_id
    end
  end
  Svc-->>MCP: CaseRunReport
  MCP-->>Ext: {total, passed, failed, bug_ids:[...]}

  Note over Ext,DB: ─ 飞轮闭环完成 ─

这一次调用让外部 Agent 拿到了：

  • 新生成的用例 ID 列表（已落库）
  • 执行结果（哪些通过、哪些失败）
  • 失败用例对应的 bug ID 列表（可直接发给开发同事）
