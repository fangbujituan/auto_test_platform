质量飞轮运转说明

一句话讲清楚： 我们围绕"需求 → 用例 → 执行 → bug → 回归"这条主线搭了一个能自转的飞轮。 每一步都封装成一个 service 函数，再通过三个出口同时暴露： HTTP 给前端、LangChain @tool 给内部 Agent、MCP Server 给外部 IDE Agent（Cursor / Claude Code 等）。 一份逻辑三个入口是这个项目最值得理解的设计。

1. 飞轮全景：闭环是怎么转的

飞轮的目标是让"被测系统的质量持续提升"自动化。下图是闭环的高层视图。 箭头方向就是数据流方向，每个节点都对应一组可独立替换的能力。

关键点：

  • 用例和 bug 都落在数据库（橙色方块），不是只在内存中流转。这样飞轮可以中断、可以人工介入、可以多端协作。
  • 失败用例自动建 bug（虚线之外这一步是自动的）。bug 的 related_test_cases 字段直接关联用例，steps_to_reproduce 抄用例的 steps —— 修复后跑回归直接拿这条 bug 关联的用例就行。
  • 每个齿轮都是独立 service，可以单独触发也可以串起来跑。

2. 三出口架构：为什么这么设计

飞轮里每一个能力（生成用例、执行用例、问答 LLM…）都同时给三种调用者用。 service 层是唯一的逻辑实现，三个出口都是它的薄包装。

这样设计的好处：

  • 逻辑只写一份。 加一个新能力，只要写一个 service 函数 + 在三个出口各包一层（每层 10 行），三种调用方式自动具备。
  • 外部 Agent 接入零成本。 同事用 Cursor 写代码时，他的 Agent 通过 MCP 直接调用我们的ai_generate_and_save_test_cases， 生成的用例自动落到我们库里。下游飞轮继续转，不需要他做任何额外集成。
  • 测试只测 service。 写 unit test 直接对着 service 写，不用过 Flask、不用启动 LangGraph、不用拉 MCP。

HTTP 出口：app/routes/ai_agent.py
@tool 出口：app/agents/tools/{testcase_tools, run_tools, case_tools}.py
MCP 出口：app/mcp_server/server.py，启动 python -m app.mcp_server

例 1 — HTTP 路由（给前端按钮用）

app/routes/ai_agent.py::RunCasesView

@ai_agent_blp.route("/run-cases")
class RunCasesView(MethodView):
    @ai_agent_blp.arguments(RunCasesRequestSchema)
    @ai_agent_blp.response(200, MessageResponseSchema)
    @login_required
    def post(self, json_data):
        from flask import g
        reporter_id = json_data.get("reporter_id")
        if reporter_id is None:
            current_user = g.get("current_user")
            if current_user:
                reporter_id = current_user.id

        report = execute_cases(                # ← 共享 service
            case_ids=json_data["case_ids"],
            project_id=json_data["project_id"],
            create_bug_on_failure=json_data.get("create_bug_on_failure", True),
            reporter_id=reporter_id,
            environment=json_data.get("environment") or "",
            version=json_data.get("version") or "",
            model=json_data.get("model"),
        )
        return jsonify({"code": 0, "message": "success", "data": report.to_dict()})

例 2 — LangChain @tool（给内部 Agent 用）

app/agents/tools/run_tools.py::run_test_cases

@tool
def run_test_cases(
    case_ids: List[int],
    project_id: int,
    create_bug_on_failure: bool = True,
    reporter_id: Optional[int] = None,
    environment: Optional[str] = None,
    version: Optional[str] = None,
    model: Optional[str] = None,
) -> dict:
    """执行一批测试用例，失败自动建 bug。

    用例由 LLM 当裁判判断 pass/fail（飞轮 v1 实现）。失败用例会自动写入
    bugs 表，并把 related_test_cases 关联回用例 id。

    Args:
        case_ids: 待执行的用例 id 列表（test_case_management.id）
        project_id: 落 bug 的项目 id
        ...
    """
    from app import create_app
    app = create_app()                       # ← 自己拉 app context
    with app.app_context():
        report = execute_cases(              # ← 共享 service
            case_ids=case_ids,
            project_id=project_id,
            create_bug_on_failure=create_bug_on_failure,
            reporter_id=reporter_id,
            environment=environment or "",
            version=version or "",
            model=model,
        )
    return report.to_dict()

4. service vs agent：别再搞混

一句话区分： service 是被调用的，agent 是会自己决定调谁的。
维度serviceagent
控制权调用方决定做什么、传什么参LLM 自己决定下一步调哪个工具
是否有 LLM 决策循环内部可能用 LLM，但只用一次（如生成用例）多轮循环：LLM → tool → LLM → tool …直到任务完成
入参函数签名固定，参数清晰通常是一段自然语言 + messages 历史
是否持有 tool自己就是被装载的工具持有一组 tool，自己挑用
行为可预测性确定性高（同样输入同样输出）同样输入可能走不同路径
测试方式单元测试函数集成测试整条 ReAct 循环
典型代码app/services/case_runner.py::execute_casesapp/agents/ui_automation_agent.py::make_agent

放到这个项目里看

execute_cases 是 service：

  • 你给它 case_ids=[12,13,14]，它就严格按这个顺序跑这三条。
  • 它内部确实调了 LLM（AIVerdictExecutor），但只是把 LLM 当确定性函数用 —— 输入用例步骤，输出 passed / failed。
  • 不会出现"LLM 觉得应该先查一下数据库再决定"这种事。

ui_automation_agent 是 agent：

  • 你给它一段话："登录系统，点击订单，截图给我"。
  • 它装载了 Playwright MCP + Chart MCP 一堆工具。
  • LLM 自己决定先调 browser_navigate，看到结果再决定调 browser_click，整个流程是 LLM 临场编排的。
  • 失败了它会自己重试、换路径。

项目里目录命名容易让人混淆，注意：
  • app/agents/tools/*.py 里的东西实际是 service 的薄包装（带 @tool 装饰器）， 它们本身不是 agent，是给 agent 用的工具。

关键约束（已校验）：

  • services/ 不依赖 agents/ 也不依赖 mcp_server/。 可以单独抽出来跑、单独单测。

两个 schemas 别搞混：

  • app/services/schemas.py ── pydantic，用于 LLM 结构化输出契约。
新增 HTTP API schema 放前者，新增 LLM 输出契约放后者。

核心设计：

  • 结构化输出契约。LLM 输出必须能解析成 GeneratedTestCaseList（pydantic schema）， 字段命名直接对齐 TestCaseManagement 表，可以一对一落库不用映射。
  • 双解析模式自动兜底。环境变量 TESTCASE_GEN_MODE=structured 走 OpenAI tool calling（GPT/Claude 这种大模型最稳）； =parser 走 prompt 注入 JSON Schema（小模型友好，是默认值）。
  • Prompt 可编辑。从数据库 ai_prompt_templates 表读，产品/测试可以在前端改提示词不需改代码。

6. 第二齿轮：从用例到 bug

读取一批用例 ID，依次执行。失败的用例自动写入 bug 表，bug 上自动关联回用例 ID。 这是飞轮闭合的关键 —— 没有这一步，前面生成多少用例都只是文档。

flowchart TB
  IN[case_ids + project_id] --> LOAD[查 test_case_management
过滤项目和有效性]

  class EXEC,V1,BUG agent
  class DB store
  class V2,V3 alt

核心设计：

  • 执行器是协议（Protocol），不是固定实现。v1 用 LLM 当裁判（最便宜，零基础设施）， 把用例的 steps + expected 喂给 LLM 让它判 pass/fail。把 service 里 AIVerdictExecutor 换成 PlaywrightExecutor 或 ApiExecutor 就升级到真实执行，编排逻辑零改动。
  • 失败即建 bug 是默认行为。reporter_id 没传则自动取一个系统用户兜底（保证飞轮不卡住）。 bug 优先级直接由用例优先级映射（P0→critical，P1→high，等等）。
  • 结果可序列化。CaseRunReport.to_dict() 直接返回给 HTTP / MCP，外部 Agent 拿到 bug_id 列表就能继续触发下游流程。
真实跑过的飞轮： 一次实测中 LLM 正确识别出"步骤是退出登录但预期是停留登录态"这种逻辑矛盾， 自动建了 bug，related_test_cases 关联到用例，actual_result 写入 LLM 观察到的 "未跳转到首页并显示用户已登录的状态"。 飞轮真转起来了。

  • 失败用例对应的 bug ID 列表（可直接发给开发同事）
